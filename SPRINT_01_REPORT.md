## Checklist results

1. **All package.json files are valid.**
   Root, `apps/web`, and `apps/api` package.json files parse correctly and
   `pnpm install` resolves every dependency with no version conflicts.

2. **pnpm workspace is configured correctly.**
   `pnpm-workspace.yaml` declares `apps/*` and `packages/*`. `pnpm install`
   at the root resolves all 3 workspace projects (root, web, api) into a
   single lockfile/node_modules tree. A stray nested `pnpm-workspace.yaml` /
   `pnpm-lock.yaml` / `node_modules` generated by `create-next-app` inside
   `apps/web` was removed so it doesn't fork the workspace.

3. **Next.js builds successfully.**
   `pnpm --filter web build` (Next.js 16.2.9, Turbopack) compiles, type-checks,
   and statically generates all routes (`/`, `/_not-found`) with no errors.

4. **Express backend starts successfully.**
   Both `pnpm --filter api dev` (tsx watch) and the compiled
   `node dist/src/index.js` start and listen on the configured port with no
   errors.

5. **Prisma schema has no errors.**
   `prisma validate` reports the schema valid. Note: this project pins
   Prisma 7, which removed inline `datasource.url` support in
   `schema.prisma` — the connection string now lives in `prisma.config.ts`
   (loaded via `dotenv/config`), and `PrismaClient` is instantiated with an
   explicit `@prisma/adapter-pg` driver adapter (`src/lib/prisma.ts`). This
   is a breaking change from Prisma 6 and earlier and was verified against
   the output of `prisma init` on this exact installed version.

6. **Tailwind is configured correctly.**
   Tailwind CSS v4 is wired through `@tailwindcss/postcss` in
   `postcss.config.mjs` and imported via `globals.css`; verified by a
   successful production build.

   Both apps use flat config (`eslint.config.mjs`): `apps/web` extends

## Known environment-specific caveat (not a project defect)

In this sandboxed container, Prisma's engine-binary download
(`binaries.prisma.sh`) was reset by the local egress proxy after ~6-7s on
every attempt, even though the same file downloaded fine via `curl` through
the same proxy. This looks like a client-timeout mismatch in
`@prisma/fetch-engine` under proxied connections, not a code defect. It was
worked around locally by pre-placing the downloaded `schema-engine` binary
in place so `prisma validate`/`generate` could run without re-downloading.
This does not affect GitHub Actions CI, which has unrestricted network
access to `binaries.prisma.sh`.
