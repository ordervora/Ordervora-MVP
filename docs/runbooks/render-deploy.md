# Deploying to Render via Blueprint (`render.yaml`)

Zero-manual-configuration deployment: the repo root's `render.yaml`
provisions the database and both services, and wires every environment
variable, migration, and health check automatically. This works entirely
from Render's mobile web dashboard — no terminal or desktop required.

## Deploy (~5 minutes, all via the Render dashboard)

1. Sign in at [dashboard.render.com](https://dashboard.render.com).
2. **New** → **Blueprint** → connect/select this repository → pick the
   branch you want deployed.
3. Render reads `render.yaml` and shows a preview of what it will create:
   one Postgres database (`ordervora-db`) and two web services
   (`ordervora-api`, `ordervora-web`). Tap **Apply**.
4. Render will ask you to confirm the paid Postgres instance's cost — this
   confirmation tap is the one step Render itself requires outside the
   Blueprint file; there's no configuration to fill in, just approval.
5. Wait for both services to build and deploy. `ordervora-api` runs
   `prisma migrate deploy` and a one-time demo seed automatically before
   it starts serving traffic (via `preDeployCommand` — see below).
6. Open `ordervora-web`'s service page for its public URL — that's your
   working deployment.

Every environment variable is wired automatically:
- `DATABASE_URL` comes from the Postgres instance Render just created.
- `FRONTEND_URL` (API) and `API_URL`/`API_HOST` (web) are cross-referenced
  between the two services' own Render-assigned hostnames — you never
  paste a URL by hand.
- `JWT_ACCESS_SECRET`/`JWT_REFRESH_SECRET` are freshly random-generated by
  Render for this deployment.
- `COMMERCE_ENCRYPTION_KEY` is a literal value baked into `render.yaml`
  (see the file's comment for why: it must be hex, and Render's
  `generateValue` produces base64). Rotate it later via
  `docs/runbooks/secret-rotation.md` if you want a deployment-specific key.

## What's automatic vs. what isn't

| | Automatic via `render.yaml` |
|---|---|
| Database provisioning | Yes |
| `prisma migrate deploy` | Yes — runs in `preDeployCommand`, before each new instance starts |
| Structural demo seed (`seed-beta.ts`) | Yes, but only once — `seed-if-empty.js` checks whether any restaurant already exists before seeding, so redeploys never duplicate data |
| Environment variables (secrets, cross-service URLs) | Yes |
| Health checks | Yes (`/health` for the API, `/` for the web app) |

**One thing this cannot automate**: the historical order volume and the
pre-placed delivery order (`seed:beta:orders`, `seed:beta:delivery-order`
from Sprint 08) drive the real HTTP checkout/order-status API, which
means they need the API already deployed and reachable — they can't run
in `preDeployCommand` (the server isn't listening yet at that point). This
is a deliberate, one-time, optional follow-up, not a missing feature:

1. Open the `ordervora-api` service page → **Shell** tab (works from an
   iPhone browser).
2. Run:
   ```sh
   BASE_URL=http://localhost:4000 node dist/scripts/seed-beta-orders.js
   BASE_URL=http://localhost:4000 node dist/scripts/demo-place-delivery-order.js
   ```
   (The Shell runs inside the same container the API is listening in, so
   `localhost:4000` reaches it directly.)

## Notes

- **Filesystem is ephemeral.** Object storage (`OBJECT_STORAGE_*` env
  vars, Phase 7) is optional and not configured by this Blueprint — image
  imports/uploads fall back to local disk, which does not persist across
  restarts or redeploys. Add an S3-compatible bucket's credentials as
  extra env vars on `ordervora-api` later if you need uploads to survive
  redeploys; no code change required (`lib/file-storage.ts` already
  supports both backends).
- **Redis is optional.** Unset, every rate limiter runs in-process
  (single-instance behavior) — fine for a single API instance. Add a
  `REDIS_URL` env var later if you scale to multiple instances.
- **Stripe is bring-your-own-provider.** There's no global demo Stripe
  key; each restaurant connects its own account from the owner dashboard's
  payments settings. Cash payment methods work immediately with no setup.
