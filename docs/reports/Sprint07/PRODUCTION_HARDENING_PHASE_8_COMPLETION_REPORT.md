# Production Hardening Phase 8 — Completion Report

**Image Optimization**

Implemented per `docs/reports/Sprint07/PRODUCTION_HARDENING_MASTER_SPEC.md`, Phase 8. Depends on Phase 7 (object storage) — every rendition generated here is written through the same `fileStorage` interface Phase 7 made durable, so this phase needed no storage-layer changes of its own.

## 1. What was done

### 1.1 `lib/image-processing.ts` — the resizer

`sharp`-based, generating a small fixed set of variants — `thumbnail` (200px), `card` (600px), `full` (1600px, capped) — per the master spec's explicit instruction to avoid "one full-resolution file reused everywhere." All variants are re-encoded to WebP (quality 80), matching what the pre-existing Performance-score suggestion text already promised callers ("AVIF/WebP renditions at every breakpoint" — `scoring/performance-score.ts`, unchanged this phase). `withoutEnlargement: true` means a small original is never upscaled. EXIF orientation is normalized (`.rotate()` with no args auto-orients from EXIF, then strips it — sharp's default) before resizing.

**Fail-open by contract** (spec item 4): `generateImageRenditions()` returns `null` — never throws — for a non-raster MIME type or for bytes sharp can't decode (corrupt/truncated upload). Callers fall back to the original file alone, exactly the pre-Phase-8 behavior.

### 1.2 Wired into the one real call site: `asset.service.ts`

`uploadAsset()` (site hero/gallery/logo/OG images — the only place in this codebase that durably stores an image for later display) now additionally calls `generateImageRenditions()` after saving the original, and — when it succeeds — saves each variant through the same `fileStorage.save()` every other write in this codebase uses, recording the resulting keys in `SiteAsset.renditions` (a `Json?` column that has existed since Sprint 06 specifically for this, per the pre-existing code comment removed by this change: *"`renditions` stays null until a future pipeline exists"*). When resizing fails, `renditions` stays `null`, unchanged from before — the Performance score and pre-publish check correctly keep reporting that asset as unprocessed rather than faking readiness, with zero changes needed to either scorer.

### 1.3 Wired into the one real read path: the renderer

`renderer/render-site.ts`'s `resolveRenderAssets()` now prefers the responsive variant sized for how each asset kind is actually displayed — hero → `full`, gallery → `card`, logo → `card` — via a new shared, dependency-free helper, `resolveAssetRenditionKey()`, added to `lib/image-processing.ts` itself (not `asset.service.ts`) specifically to avoid a circular import (`site.service.ts` → `renderer/render-site.ts` → `asset.service.ts` → `site.service.ts`, discovered and avoided during implementation — see §3). It falls back to the full-resolution original when `renditions` is `null`, so an asset uploaded before this phase (or one whose resize failed open) keeps rendering exactly as it did before.

### 1.4 Import adapters — audited, no relevant call site (documented, not silently skipped)

The master spec's "Affected files" list for this phase includes `apps/api/src/modules/imports/**` adapters "where relevant." Tracing every adapter (`pdf.adapter.ts`, `image.adapter.ts`, `website.adapter.ts`, `google-maps.adapter.ts`) confirms none of them persist images at all — every one only holds image bytes transiently, in memory, to pass to `extractMenuFromImages()`/vision extraction, then discards them. There is no `fileStorage` call anywhere under `modules/imports/` (confirmed via `grep`) and no per-menu-item photo field on the `MenuItem` model — imported data is category/item name/description/price/`businessProfile` text only. **There is genuinely no relevant call site here** — wiring a resizer into a code path that never stores an image would be dead code, not a real integration, so none was added. This is a scope finding, not a shortcut: flagged explicitly rather than silently claimed as "done."

### 1.5 `apps/web` audit — no image-rendering component exists yet to point at the new variants

Audited (per spec item 3) rather than assumed. The published, publicly-served site pages are static HTML generated by `apps/api`'s own renderer (§1.3 above) — not React/Next.js — so there is no `apps/web` component in that path at all. Within `apps/web` itself, the only `next/image`/`<img>` usage found is in `apps/web/src/app/page.tsx`, which is the unmodified Next.js starter template (`/next.svg`, `/vercel.svg`) — unrelated to `SiteAsset`. No dashboard page in `apps/web/src/app/dashboard/website/**` currently calls the asset-upload endpoint or renders a `SiteAsset` at all (`storageKey`/`SiteAsset` do not appear anywhere under `apps/web/src`, confirmed via `grep`). This is a pre-existing gap unrelated to this phase — not something this phase's instructions authorize fixing (**"Do NOT modify unrelated application business logic"**) — so it is reported as an honest audit finding: there is currently no `apps/web` component that needs pointing at the new variants, because no `apps/web` component consumes `SiteAsset` images yet.

## 3. A design decision made during implementation: avoiding a circular import

The natural place for a "given a `SiteAsset`, pick the right rendition key" helper is next to `AssetRenditions`' producer, `asset.service.ts`. But `site.service.ts` already imports `renderer/render-site.ts` (to call `renderAllPages` on publish), and `asset.service.ts` imports `site.service.ts` (`getOwnSiteById`) — so `render-site.ts` importing from `asset.service.ts` would close a cycle: `site.service.ts` → `render-site.ts` → `asset.service.ts` → `site.service.ts`. Resolved by keeping `AssetRenditions` and `resolveAssetRenditionKey()` in `lib/image-processing.ts` — a dependency-free pure module already sitting below both `asset.service.ts` (write side) and `render-site.ts` (read side) in the dependency graph, so both can import it without either importing the other's module.

## 5. Known limitations

- No `apps/web` UI currently exists to exercise a real end-user upload → view round trip (§1.5) — the interface-level round trip (§4) is the honest substitute available in this environment.
- `sharp` ships platform-specific prebuilt `libvips` binaries as optional dependencies; the Alpine-based `Dockerfile` (Phase 4) installs via a plain `pnpm install` with no `--ignore-optional` flag, which should resolve the correct `linuxmusl-x64` binary automatically — this was not exercised inside an actual container build in this sandbox (Docker Hub image-layer pulls remain blocked here, the same carried-forward limitation as Phases 4–7). Confirmed instead by running the resizer directly under this sandbox's own Node 22 install (`cd apps/api && node -e "require('sharp')"` and the full `image-processing.test.ts` suite), which is the same host architecture (`linux-x64`) the Alpine image's musl variant targets, though not bit-for-bit the same libc.
- Real image processing was already flagged as a known gap in `asset.service.ts`'s own pre-existing code comment ("Real image processing... is NOT implemented") — that comment is now removed/updated since it describes a state this phase resolves.
